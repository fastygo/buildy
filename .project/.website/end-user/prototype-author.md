# End User: Prototype Author

## Profile

| Field | Detail |
|-------|--------|
| Archetype | Person who turns ideas into reviewable prototypes |
| Typical titles | Product Manager, Founder, Product Designer, Agency PM, Design Engineer |
| Technical level | Low to medium — comfortable with concepts, not daily Go coding |
| Session length | 30–90 minutes; returns across days until preview approved |
| Success feeling | Stakeholders say "yes, this is what we meant" |

## Primary workflow

```text
New project → pick SaaS landing template → prompt/describe product
    → adjust pages/blocks/fields → preview in workspace
    → share link → collect feedback → iterate PromptRun
    → request export when approved
```

## Tasks in BuildY

| Task | Workspace surface |
|------|-------------------|
| Define product narrative | Prompt + project metadata |
| Map landing sections | Page / Block / FieldMapping editor |
| Validate flow | Preview workspace (navigation, empty states) |
| Compare iterations | PromptRun history, diffs (later) |
| Hand off | Flag "ready for export" for developer |

## Motivations

- **Speed** without begging eng for spike  
- **Clarity** — structure forces thinking about content types  
- **Credibility** — live preview beats slide deck  
- **Continuity** — confidence export won't be discarded  

## Frustrations to avoid in UX

- Browser feels like IDE (too many files, syntax noise)  
- Opaque codegen with no preview  
- Hardcoded copy with no CMS path  
- Losing work between sessions  
- Unclear "what changed" after prompt  

## UX patterns (category)

Common AI-builder patterns worth matching in workspace UX:

- Chat-first project creation
- Real-time preview feel
- Template gallery entry point
- Visual refinement for layout/copy (within block model)

## BuildY-specific behaviors to emphasize

- **Schema panel** alongside preview — "what CMS will store"  
- **Generated files** as read-only review tab, not primary editor  
- **Block-to-field** mapping visible for marketing handoff
