# Ideal Customer Profile (ICP) — Overview

An ICP describes the **company or organization** that gets disproportionate value from BuildY — not an individual user.

## ICP selection criteria

| Dimension | Strong fit | Weak fit |
|-----------|------------|----------|
| Stage | Pre-seed to Series B; agency with 3–30 delivery staff | Enterprise with mandated non-Go stack |
| Team | 0–15 engineers; product/marketing close to delivery | Large IT procurement, 6-month RFP cycles |
| Stack posture | Open to or already on FastyGo; Go-friendly | Committed Next/Supabase only |
| Use case | SaaS landing, product marketing site, CMS-backed MVP | Mobile-first consumer app, game, heavy realtime |
| Workflow | Prototype → stakeholder review → engineering extend | Buy finished hosted app, no code touch |
| Content need | Marketing edits copy/blocks post-launch | One-shot campaign page, discard after |

## ICP vs persona

```text
ICP (company)     →  "Who is the account?"
Buyer persona     →  "Who signs or approves?"
End user          →  "Who uses BuildY weekly?"
```

One ICP typically maps to 2–3 buyer personas and 2–4 end users.

## Revenue and expansion signals (future)

- Multiple projects per workspace  
- GitHub sync and team seats  
- AppCMS live instances connected to BuildY projects  
- Template/pattern library usage across clients  

Billing and GitHub App complexity are **post-MVP**; ICPs should still be chosen for landing + export fit today.
